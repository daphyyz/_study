## 1. MariaDB vs MySQL: 형제에서 경쟁자로
* **역사적 관계와 철학:** MySQL이 오라클에 인수된 후, 오픈 소스의 자유를 지키기 위해 원개발자가 포크하여 탄생했습니다. 초기엔 100% 호환되었으나 현재는 MariaDB가 독자적인 기능을 대거 추가하며 차별화된 노선을 걷고 있습니다.
* **라이선스와 커뮤니티:** MySQL은 오라클의 상용 정책에 따라 일부 고급 기능이 유료(Enterprise)로 제한되지만, MariaDB는 거의 모든 기능을 오픈 소스(GPLv2)로 제공하여 커뮤니티의 강력한 지지를 받습니다.

## 2. MariaDB만의 독보적 기능 (MySQL과의 차이점)
* **Sequence (시퀀스):** 특정 테이블에 종속된 `AUTO_INCREMENT`와 달리, DB 내 독립적인 객체로 존재합니다. 여러 테이블이 하나의 순번을 공유하거나 데이터를 삽입하기 전에 번호를 미리 선점할 수 있어 복잡한 비즈니스 로직 구현에 유리합니다.
* **System-Versioned Tables (이력 관리):** 테이블 생성 시 옵션을 주면 데이터의 모든 변경 이력을 DB가 숨겨진 타임스탬프 컬럼과 함께 자동 관리합니다. 별도의 이력 테이블 설계 없이도 특정 시점의 과거 데이터를 조회하거나 전체 변경 로그를 확인하는 '시간 여행 쿼리'가 가능합니다.
* **MyRocks & Spider 엔진:** 쓰기가 극도로 많은 환경을 위해 페이스북이 개발한 LSM-Tree 기반의 MyRocks와, 물리적으로 떨어진 여러 서버의 테이블을 하나처럼 묶어 샤딩(Sharding)을 구현하는 Spider 엔진을 기본으로 탑재하고 있습니다.
* **Oracle Compatibility:** 오라클의 저장 프로시저 문법인 PL/SQL을 거의 그대로 실행할 수 있는 호환 모드를 지원하여, 엔터프라이즈 환경에서 오라클 DB를 오픈 소스로 전환할 때 발생하는 비용을 획기적으로 낮춰줍니다.

## 3. 스토리지 엔진 내부 원리
* **LSM-Tree (Log-Structured Merge-Tree):** 기존 데이터를 수정하지 않고 항상 새로운 파일 끝에 이어 붙이는 '순차 쓰기' 방식을 사용합니다. 디스크 헤더가 여기저기 점프하는 랜덤 쓰기 부하를 없애기 때문에, 데이터가 쏟아져 들어오는 쓰기 집중형 서비스에서 압도적인 성능을 냅니다.
* **Compaction (데이터 병합):** LSM-Tree의 특성상 발생하는 중복 데이터를 백그라운드에서 정리하는 작업입니다. 여러 파일에 흩어진 동일 키값 중 최신본만 남기고 정렬하는 과정으로, 비싼 작업이지만 시스템 부하를 고려해 야금야금 수행하며 저장 공간을 확보합니다.
* **Bloom Filter (확률적 문지기):** 아주 적은 메모리(비트 배열)만 사용하여 특정 데이터가 디스크 파일 안에 있는지 미리 검사합니다. "절대 없음"은 100% 정확하게 판별하여 불필요한 디스크 읽기를 차단하며, "있을 확률"이 높은 경우에만 실제 파일을 뒤져 읽기 성능을 보완합니다.

---

## 4. 대규모 분산 아키텍처
* **CQRS (명령과 조회의 분리):** 데이터의 상태를 바꾸는 '명령(Command)'과 데이터를 보여주는 '조회(Query)'의 경로를 물리적으로 나누는 전략입니다. 쓰기엔 MariaDB를, 읽기엔 Redis나 Elasticsearch를 사용하여 각 작업에 최적화된 성능과 안정성을 확보합니다.
* **최종 일관성 (Eventual Consistency):** 전 세계 사용자가 동시에 이용하는 서비스(유튜브 조회수 등)에서 모든 데이터를 실시간으로 완벽하게 맞추는 대신, 시스템 안정성을 위해 약간의 지연 후 결국 일치하게 만드는 설계 철학입니다.
* **Kafka (메시지 브로커):** 쓰기 DB의 변경 사항을 읽기 DB로 전달하는 거대한 데이터 고속도로 역할을 합니다. 대량의 이벤트를 순차 로그로 쌓아두었다가 소비자가 감당할 수 있는 속도로 처리하게 하여 시스템 간 결합도를 낮추고 유실을 방지합니다.
